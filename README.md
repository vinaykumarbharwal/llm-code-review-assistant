
# LLM Code Review Assistant
A concise, practical description and how-to for developers using the repository.

This project provides a backend service that analyzes Git diffs or pull requests and returns structured code-review feedback generated by an LLM. It is intentionally backend-first so it can be integrated into CI/CD, GitHub webhooks, or a web frontend.

Quick summary
---------------
- Backend: FastAPI application in `src/app` exposing `/review`, `/ingest`, `/webhook/github`, and `/health`.
- Async processing: Celery worker `src/app/workers/review_worker.py` with Redis broker.
- RAG placeholders: `src/app/rag/` contains embedding/retrieval stubs.
- LLM integration: `src/app/core/llm_client.py` contains a stub; replace with Groq or other SDK to get real reviews.
- Frontend: Minimal React app in `frontend/` that posts diffs to the `/review` endpoint.

Requirements
------------
- Python 3.11+
- Node.js 16+ (for the frontend)
- PostgreSQL (or cloud Postgres like Neon) for storing reviews
- Redis for Celery broker

Environment variables
---------------------
Copy `.env.example` to `.env` and set values. Key variables used by the app:

- `GROQ_API_KEY` — your Groq (or placeholder) API key for LLM calls
- `GITHUB_WEBHOOK_SECRET` — secret to validate GitHub webhooks
- `DATABASE_URL` — SQLAlchemy async URL, e.g. `postgresql+asyncpg://user:pass@host:port/dbname?sslmode=require`
- `REDIS_URL` — e.g. `redis://localhost:6379/0` or a cloud Redis URL
- `LLM_MODEL` — model id (e.g. `llama3-70b-8192`) used by your LLM client

Local setup (backend)
---------------------
1. Create and activate a Python virtual environment:

```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1   # Windows PowerShell
# or on macOS/Linux:
# source .venv/bin/activate
```

2. Install Python dependencies (project uses PEP 621 `pyproject.toml`):

```powershell
pip install uv
uv pip install -e .[dev]
```

3. Ensure your database is available and `DATABASE_URL` is set. For Neon or cloud Postgres, use the provided connection string but change the scheme to `postgresql+asyncpg://` when using async SQLAlchemy.

4. Start the FastAPI app:

```powershell
uvicorn src.app.main:app --reload
```

5. Start a Celery worker (in another terminal):

```powershell
celery -A src.app.workers.review_worker worker --loglevel=info
```

Local setup (frontend)
----------------------
1. Change into the `frontend` folder:

```powershell
cd frontend
```

2. Install dependencies and start the dev server:

```powershell
npm install
npm start
```

3. Open `http://localhost:3000` and paste a unified diff, then click *Review*.

API endpoints (useful examples)
--------------------------------
- `GET /health` — quick liveness check
- `POST /review` — submit `{ "diff": "<unified diff text>" }` and receive a `ReviewResponse` JSON (schema in `src/app/schemas/review.py`).
- `POST /ingest` — upload a guidelines document for RAG-ingestion (placeholder)
- `POST /webhook/github` — GitHub webhook receiver with signature validation (placeholder)

Example curl request (Unix/macOS):

```bash
curl -X POST http://localhost:8000/review \
	-H 'Content-Type: application/json' \
	-d '{"diff":"<paste-unified-diff-here>"}'
```

Example PowerShell request:

```powershell
$diff = @'
<paste-unified-diff-here>
'@

Invoke-RestMethod -Uri http://localhost:8000/review -Method Post -ContentType 'application/json' -Body (@{ diff = $diff } | ConvertTo-Json)
```

Project layout (where to look)
-----------------------------
- `src/app/main.py` — app factory and CORS middleware
- `src/app/api/` — `review.py`, `ingest.py`, `webhook.py` (API endpoints)
- `src/app/core/` — `diff_parser.py`, `prompt_builder.py`, `llm_client.py`, `response_parser.py` (core logic)
- `src/app/rag/` — `embedder.py`, `retriever.py` (RAG stubs)
- `src/app/github/` — `client.py`, `webhook.py` (GitHub helpers)
- `src/app/models/db.py` — SQLAlchemy ORM model for reviews
- `src/app/schemas/review.py` — Pydantic response/request schemas
- `src/app/workers/review_worker.py` — Celery task that runs reviews
- `frontend/` — minimal React UI that posts diffs to `/review`

How the review flow works (high level)
-------------------------------------
1. User submits a diff (API or frontend) to `/review`.
2. The request is handled by FastAPI; long-running work is queued to Celery.
3. Worker parses the diff (`unidiff`), optionally retrieves relevant guidelines from ChromaDB, builds a prompt, and calls the LLM via `src/app/core/llm_client.py`.
4. The LLM response is parsed and shaped into the `ReviewResponse` schema and stored in PostgreSQL.
5. Optionally the worker posts inline comments back to GitHub using `PyGithub`.

Troubleshooting
---------------
- `Error: Failed to fetch` in the frontend: enable CORS (already added to `main.py`) and ensure backend is running on `http://localhost:8000`.
- Celery cannot connect to Redis: ensure `redis-server` is running or set `REDIS_URL` to a cloud Redis.
- `react-scripts` not recognized: run `npm install` in `frontend` and use Node 16+.
- Database connection issues: verify `DATABASE_URL` and that the Postgres instance accepts connections from your machine (Neon or cloud services may require SSL or a specific host allowlist).
